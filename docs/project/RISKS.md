# 喵懂（喵测）项目风险清单与应对预案（RISKS）

版本：v0.1（适配 8 周 MVP；1-2 人轻团队）  
目标：把“最可能翻车/最伤口碑/最卡工期”的风险前置，用**灰度+回滚+口径**兜底。

---

## 1) 风险分级与触发机制

| 等级 | 定义 | 处理要求 |
|---|---|---|
| P0（致命） | 可能导致大面积负面口碑、合规事故、无法上线 | 必须有预案；必须可灰度/可回滚；上线前演练 |
| P1（高） | 影响核心指标（留存/复测/分享）或显著拖慢工期 | 需监控指标；有替代方案与降级 |
| P2（中） | 局部体验/局部模块延期 | 可接受阶段性欠债，但要记录与排期 |

触发机制（最小化）：
- 每次发布生成 `content_version`，灰度 5% 起步  
- 观察窗口：发布后 **24h/72h**（看板对比）  
- 触发回滚条件（示例，可按实际调整）：  
  - 推理成功率下降 ≥ 5%（绝对值）  
  - problem 有用率下降 ≥ 10%（相对值）  
  - 负面反馈密度（helpful=false）显著上升

## 3) 关键风险预案（逐条可执行）

### R-01：准确性争议（P0）

**风险本质**：用户把“意图标签”当成“准确翻译”，期待落空→嘲讽/差评→传播负面。

**预防（上线前必须具备）**
1. **产品口径硬约束**：全站只说“意图/倾向/可能”，不说“猫在说：xxx句子”。  
2. **可解释性必须出现**：每次结果至少 1 条“证据解释”（PATTERN/CONTEXT）。  
3. **复测机制前置**：结果页给“72小时复测”入口，强调“看趋势更可靠”。  
4. **分享卡规则**：只分享 A 类意图 + 证据解释；B 类风险徽标 `share_safe=false` 永不出现在分享卡上。

**监控（上线后）**
- helpful=false 的反馈中，包含“骗/不准/玄学/翻译”关键词比例
- 分享率 vs 有用率：如果分享高但有用率低，说明“传播有效但价值不稳”

**应对（触发后）**
- 立即将解释文案与免责声明加强（配置化发布，content_version+1，灰度）
- 若仍恶化：回滚到上一个 content_version，并下线争议模板

---

### R-02：健康提示合规风险（P0）

**预防**
- 风险分级只给“行动建议”，不下疾病结论  
- 全局免责声明统一来源（后台可配置，避免口径漂移）  
- 红色提示只在“安全提醒”级别出现（如便血/尿血/持续尖叫等），并建议就医

**应对**
- 发现不当口径：第一时间发布配置修正（灰度→全量）
- 必要时临时关闭某些 risk_badge 输出（feature flag）

---

### R-03：冷启动不出圈（P0）

**预防**
- 保证“可晒、可比、可连载”至少三件事：  
  1) 结果卡可晒（含证据解释）  
  2) 复测对比卡可晒（前后对比）  
  3) 成就系统最小可用（驱动连载）  
- 人格卡优先级高，但工程顺序可后于成就（成本/依赖）

**应对**
- 一周复盘：若收藏/评论高但回流低 → 优化“发帖文案模板+私信SOP”
- 若曝光低 → 调整内容选题到“新人强痛点”与“复测对比挑战”

---

### R-04：无灰度/回滚导致无法止损（P0）

**预防**
- 发布中心（content_version + rollout_percent + rollback）必须在 W7 前可用  
- 发布演练（灰度→回滚→复盘）必须完成（W7-W8）

**应对**
- 一旦出现指标明显下滑：立即回滚到上一版本

---

### R-05：音频兼容性问题（P1）

**预防**
- 客户端录音优先 m4a(AAC-LC)，10秒上限  
- 服务端统一转码到 16kHz/mono/PCM WAV  
- 弱网重试与上传失败提示（明确下一步）

**应对**
- 机型/系统版本维度定位失败原因（埋点：content_type/network_type/error_code）
- 临时降级：提示“建议在安静环境录3-5秒”并提供示例

---

### R-06：推理服务不稳定/超时（P1）

**预防**
- Go API 对推理请求做超时（3s）与熔断；失败返回 `degraded=true` 结构  
- 推理服务可水平扩容；并发隔离，避免拖垮业务API

**应对**
- 启用更保守的超时/限流；高峰期提示“稍后再试”
- 必要时切换到“仅问题库建议”模式（feature flag）

---

### R-07：运营后台误改内容（P1）

**预防**
- 草稿/发布状态机必须有  
- 发布前预览 + 差异摘要（至少文本版）  
- 操作日志（谁在何时改了什么）

**应对**
- 发现误改：发布修复版本；或直接回滚

---

### R-08：商业化引发反感（P1）

**预防**
- 先C后A：口碑期 buy_link 永远为 null  
- 工具区默认折叠；每场景≤2个“省力工具”；必须提供“不买也能做”的替代方案

**应对**
- 若工具区展开率高但有用率低/负反馈高：更克制露出，先优化内容而不是加SKU

---

### R-09：数据闭环缺失（P1）

**预防**
- P0 事件必须齐全：推理成功、结果页、追问、建议展开、反馈、复测、分享、工具区展开  
- 指标口径固定写入 `ANALYTICS_SPEC.md` 并落到看板

**应对**
- 若发现缺失：优先补埋点与口径一致性，再谈优化策略

---

### R-10：范围失控（P1）

**预防**
- 同城/同龄对比（增长模块#5）明确后置（样本量与合规充分后再启用）  
- 条件引擎/复杂个性化在 MVP 只留字段，不做复杂规则执行

**应对**
- 每周范围审查：所有新增需求必须指向“复测率/有用率/分享率”三者之一

---

## 4) 上线前“必做清单”（Gate）

- [ ] 发布中心可用：content_version + 灰度 + 回滚  
- [ ] 发布演练完成：5%灰度 → 观察指标 → 回滚 → 复盘记录  
- [ ] 分享卡不包含任何 B 类风险内容（share_safe=false）  
- [ ] 全局免责声明口径统一且可配置  
- [ ] 推理失败有降级：degraded=true + 引导问题库  
- [ ] 小红书 SOP 可执行（评论/私信/授权口径）
