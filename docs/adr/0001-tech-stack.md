# ADR-0001：技术栈选择（Flutter + Go API + Python 推理）

日期：2026-04-22  
状态：Accepted

## 背景

产品目标为 8 周内可上线测试，并且架构需可平滑扩展到 10 万 DAU；同时 AI 推理与内容策略迭代频繁，需要工程上高内聚、低耦合、可维护。

## 决策

- 客户端：Flutter（iOS/Android 一套代码，体验一致）
- 业务 API / BFF：Go（承载并发与稳定性；工程治理成熟）
- AI 推理：Python（特征/模型迭代效率最高）
- 存储：Postgres（权威数据）、Redis（缓存/限流）、对象存储（音频）

## 备选方案

1) 双端原生（Swift/Kotlin）：质量高但成本高，不利于 8 周 MVP  
2) Node/TS 业务 API：迭代快但高并发/工程治理需额外投入；推理仍需 Python  
3) 端侧推理：隐私强但模型/包体/适配成本高，上线慢

## 后果

### 正向
- 业务与推理解耦：推理升级不影响客户端，客户端体验迭代不影响模型服务
- 可扩展：推理服务可横向扩容；API 具备限流/熔断/超时能力

### 负向
- 需要跨语言协作（Go↔Python），需定义稳定的接口契约与版本策略
