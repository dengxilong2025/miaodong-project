# 技术架构设计（MVP→10万DAU准备）

目标：**按 10 万 DAU 准备、按 1 万 DAU 成本运行**；模块化、解耦、高内聚、可扩展、可维护。

---

## 1. 总体分层与分域

### 1.1 客户端（Flutter）

只负责体验与分发：
- 录音、弱网重试、上传
- 结果渲染、分享物料生成
- 本地缓存（最近记录）
- 埋点与配置拉取

不承载核心推理逻辑（避免模型升级困难/包体膨胀/规则散落）。

### 1.2 云端（拆 3 个域，边界稳定）

1) **业务 API / BFF（Go）**  
用户、猫档案、问题库/内容库、运营配置、推理编排、推荐策略、会员、灰度/AB。

2) **AI 推理服务（Python）**  
音频→特征/模型→意图标签+置信度+解释因子+B风险徽标。

3) **数据与增长（Data）**  
埋点、漏斗、内容效果、推荐效果、训练数据闭环（可异步化）。

---

## 2. 技术栈选择（摘要）

- Flutter：统一 iOS/Android 体验与交付效率
- Go（业务 API）：承载并发、稳定性与工程治理（限流/熔断/超时/追踪）
- Python（推理）：承载模型与特征迭代速度
- Postgres：权威业务数据与内容数据（可审计、可关联、可后台化）
- Redis：缓存、限流计数、灰度/配置缓存
- 对象存储：音频上传与存储（直传，减轻 API 压力）

---

## 3. 关键链路（同步推理）

1) 客户端获取上传 URL → 直传对象存储  
2) 客户端 `POST /v1/inference` 提交 audio_id + 场景问答 + pet_id  
3) Go API 执行推理编排：拉取/校验音频 → 调用 Python 推理 → 合并内容策略  
4) 返回：主意图 + 置信度 + 解释因子 + 风险徽标 + 追问 + 建议 + 工具区（先C）+ 分享模板

**降级策略（必须）**
- 推理失败：返回“仅问题库建议 + 安抚文案 + 再试提示”
- 超时：提示稍后再试；必要时可引入排队/异步 job（后续）

---

## 4. 并发与容量预估（10万DAU）

保守估算：10万DAU × 2次/日 ≈ 20万推理/日  
平均 QPS ≈ 2-3；晚高峰按 10×：20-30 QPS。

关键原则：
- 上传走对象存储直传（避免 API 机器带宽/磁盘瓶颈）
- 推理服务可横向扩容；与业务 API 做并发隔离
- Go API 做限流/熔断/超时，避免推理拖垮全站

---

## 5. 内容与配置（运营可后台改）

采用 “**Postgres 权威内容** + **运行时配置（JSON策略层）**”：

- 权威内容（可检索/关联/统计/审计）：问题库、追问、建议、复测方案、风险分级口径、工具区清单（先C）
- 策略配置（高频变更/灰度/AB）：解释文案模板、分享模板、建议排序与展示策略

发布要求：版本化（content_version）+ 灰度 + 回滚。

---

## 6. 安全与合规（摘要）

- 非诊断声明、风险分级口径统一
- 音频默认短期保存（如 7 天可配置）+ 用户删除入口
- 最小化采集：只采集对解释/画像必要的信息
